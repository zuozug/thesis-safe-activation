# 第三章 非线性函数替代模块设计

## 3.1 模块总体设计目标

本模块以“可插拔、可切换、可复用、面向 HE/MPC 兼容”为目标展开设计。所谓可插拔，是指模块能够直接嵌入 PyTorch 模型，并替换现有激活层；所谓可切换，是指模块能够在原函数与近似函数之间快速切换；所谓可复用，是指不同非线性函数共享统一接口与配置方式；所谓面向 HE/MPC 兼容，则是指替代函数尽量仅由加法和乘法构成，为后续接入隐私计算框架提供结构基础。

### 3.1.1 设计原则

1. 统一接口；
2. 最小侵入；
3. 易于实验对照；
4. 便于可视化分析；
5. 支持未来扩展到 HE/MPC 场景。

### 3.1.2 模块功能要求

- 支持原函数计算；
- 支持近似函数计算；
- 支持参数化配置阶数与区间；
- 支持自动替换模型中的激活层；
- 支持输出实验所需的误差统计。

## 3.2 原函数与近似函数接口设计

### 3.2.1 接口设计目标

为了降低不同函数实现之间的差异，本文将原函数与近似函数抽象为统一接口，使实验脚本、可视化脚本和模型替换逻辑能够复用。

### 3.2.2 基础接口形式

- `f(x)`：原始非线性函数；
- `approx_f(x)`：近似替代函数；
- `build_runtime_config()`：构造运行时配置；
- `get_default_degree()`、`get_default_interval()`、`get_default_method()`：读取默认配置。

### 3.2.3 输入输出约定

模块的输入为 PyTorch Tensor，输出与输入 shape 保持一致。对于 Softmax，输出需额外满足概率分布约束，即各分量非负且在指定维度上求和接近 1。

## 3.3 四类函数的近似方案设计

### 3.3.1 ReLU 近似设计

ReLU 是本课题的核心主实验对象。由于其在 \(x=0\) 处不可导，因此不适合直接使用局部泰勒展开。本文优先采用 Chebyshev 或最小二乘全局逼近，比较 2 / 4 / 6 阶多项式的效果，并默认将逼近区间设为 `[-3, 3]`。

### 3.3.2 Sigmoid 近似设计

Sigmoid 是典型平滑函数，理论上适合泰勒展开与最小二乘拟合。但其饱和区误差需要重点分析。本文将其作为补充函数，比较 3 / 5 / 7 阶近似方案，默认区间取 `[-4, 4]`。

### 3.3.3 GELU 近似设计

GELU 是第二主实验对象，其表达形式比 ReLU 更复杂，更能体现模块对现代激活函数的支持能力。本文采用 Chebyshev 或最小二乘方法对其进行 3 / 5 / 7 阶近似，并将默认区间设为 `[-3, 3]`。

### 3.3.4 Softmax 近似设计

Softmax 是最复杂的近似对象。其核心困难在于指数函数与归一化结构。本文采用“先对指数函数进行多项式近似，再做稳定化归一化”的策略，将其作为补充实验对象，重点验证概率分布性质和数值稳定性，而不将其作为主训练主线。

## 3.4 PyTorch 模块封装设计

### 3.4.1 模块类设计

本文设计了四个模块类：

- `ApproxReLU`
- `ApproxSigmoid`
- `ApproxGELU`
- `ApproxSoftmax`

它们共同继承统一的激活模块基类，并支持 `exact` / `approx` 两种运行模式。

### 3.4.2 模块封装要求

- 兼容 `nn.Module`；
- 支持前向传播；
- 支持切换 exact / approx 模式；
- 支持 degree / interval / method 等参数配置。

### 3.4.3 前向逻辑说明

在 `exact` 模式下，模块直接调用原函数实现；在 `approx` 模式下，模块调用多项式近似函数。对于 Softmax，还需要显式指定作用维度并保证数值稳定性。

## 3.5 自动替换机制设计

### 3.5.1 替换目标

为了减少人工修改模型结构的负担，本文实现了激活层自动替换机制，将原始模型中的 `nn.ReLU`、`nn.GELU`、`nn.Sigmoid`、`nn.Softmax` 自动替换为对应的 Approx 模块。

### 3.5.2 替换逻辑

自动替换机制主要包括以下步骤：

1. 遍历模型子模块；
2. 识别待替换激活层；
3. 构造对应 Approx 模块；
4. 保留必要参数（如 Softmax 的维度）；
5. 输出替换记录。

### 3.5.3 验证方式

自动替换完成后，可通过以下方式验证：

- 替换前后模型结构打印对照；
- 前向传播是否成功；
- 输出 shape 是否一致；
- 替换记录是否完整。

## 3.6 面向 HE/MPC 的兼容性说明

### 3.6.1 兼容性来源

本文的近似函数主要基于多项式表达，核心运算由加法与乘法组成，因此在结构上更适合后续与 HE/MPC 场景对接。

### 3.6.2 本文边界

本文不实现完整 HE/MPC 推理系统，而是在明文环境下验证替代模块的正确性、可用性与性能影响。

### 3.6.3 后续扩展方向

后续可从以下几个方向扩展：

- 接入 TenSEAL / CrypTen 等框架；
- 尝试 LUT 与分段多项式策略；
- 扩展到更复杂模型，如 Transformer 结构。
